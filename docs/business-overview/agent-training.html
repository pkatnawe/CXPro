Internal · grounding & training plan · how the role agents learn the work

How we train the role agents — grounded in real documents, cited, and never guessing.

The plan to turn the commissioning body of knowledge and each project's documents into per-role AI agents that draft the work and cite every claim. It's built on what already ships — DSPy signatures, a pgvector store, and a refuse-when-unsure gate in cx_execution_agent — and extends it to a tiered knowledge corpus, role-scoped retrieval, evaluation, and a licensing-clean data strategy. Pairs with the Study Guide (the knowledge) and AI in CXPro (the choreography). Strictly internal — see the folder README.

The grounding contract

Four rules every agent obeys — they're the product's trust layer

These aren't aspirations; three of the four are already enforced in code. The plan is to hold the line as we add knowledge and roles.

🔎

Retrieval-grounded, not memorized

Agents answer from retrieved documents, not from model weights. The knowledge lives in a versioned corpus we control, so we can update it, audit it, and trust it — and we never depend on what a foundation model happened to absorb in training. RAG over fine-tuning, for grounding.

🧾

Cited, or it doesn't ship

Every generated step carries a citation to the chunk it came from — already enforced by the DSPy signatures (GenerateL2ChecklistSignature) with retry-until-cited logic. No citation, no output.

🛑

Refuse when unsure

If retrieval returns too few or low-confidence chunks, the agent refuses and emits an AIRefusal event rather than hallucinating — the confidence gate already in cx_execution_agent (needs ≥3 chunks, ≥2 high-confidence). A confident wrong answer is the only unacceptable outcome.

🙋

AI drafts, humans decide

Agents prepare work — procedures, triage, briefs — but never change project state on their own. A human confirms every proposal, and that confirmation is logged and becomes training signal. The role agents inherit this contract unchanged.

Step 1 · the corpus

Three layers of knowledge — global, project, and learned

An agent's answer is only as good as what it can retrieve. We ground on three distinct layers, each with different ownership, licensing, and update cadence. Today the pipeline ingests Layer B (project docs) only; the plan adds A and C.

Layer A · global
Commissioning body of knowledge
The durable, org-wide knowledge: the Cx process, standards, sample tests, sequences. Mostly public-domain federal/lab sources. Same for every customer.
Layer B · project
This project's documents
Specs, submittals, drawings, sequences of operation — already detected, chunked, embedded, and stored in document_chunks. Customer-owned context.
Layer C · learned
Org-scoped lessons
Human edits, resolved deviations, accepted/rejected drafts — captured and fed back so each project makes the next one smarter. The moat.
Retrieval blends all three: the global process tells the agent how commissioning works, the project docs tell it what this building is, and the lessons tell it what this org has learned.

Why the split matters: Layer A is written once and shared across all customers (and is where the free authoritative sources go). Layer B is per-project and already flows through ingestion_pipeline. Layer C is the compounding asset — it's also why pricing is per-org with org-scoped lessons (see Pricing). Keeping them as separate, tagged corpora lets retrieval boost the right layer per question and keeps licensing clean (next section).

Step 2 · the role agents

Each role gets a scoped agent — same engine, different lens

Today there's one generic cx_execution agent. The plan: one persona per role, each with a tailored system signature, a scoped retrieval filter (so the field tech isn't searching the whole spec set), an explicit set of draft-only actions, and the shared refuse/cite guardrails. Step 3 · the pipeline

The grounding pipeline — and the gaps to close first

The skeleton exists (detect → chunk → embed → store in pgvector → retrieve → generate-with-citations → refuse-or-draft → human review). The plan hardens it for a multi-source corpus and fixes the issues the architecture review surfaced.

Tag every chunk with a source tier

Add authority metadata — standard · federal/public-domain · project · vendor · lesson — to each chunk. Drives both citation-trust display (we already grade sources on this site) and retrieval boosting, and makes licensing enforceable at query time.

Chunk by structure, not by 500 characters

Replace fixed 500-char windows with structure-aware chunking that respects headings, tables, and list boundaries. Commissioning specs and sequences are highly structured; mid-table splits destroy retrievability.

Fix the embedding path

Two concrete bugs from the review: the document_chunks schema declares vector(1536) while Gemini text-embedding-004 emits 768 dims (reconcile the schema), and retrieval currently searches with the raw query text instead of an embedded query — embed the query and do true cosine search.

Retrieve with a role-scoped filter

Before similarity search, filter by role · discipline · corpus layer · project, then boost by source tier. The field tech's query never ranks the whole spec set; the OCA's can.

Generate through a role signature, keep the gates

Extend the DSPy signature pattern per role; keep mandatory citations and the ≥3-chunk / ≥2-high-confidence refusal gate. Output stays a draft + citations written to the existing citations table.

Version the corpus and capture the edit

Stamp each draft with the corpus version and chunk IDs that produced it (audit + reproducibility). When the human accepts or edits, capture the delta into Layer C — closing the loop the review flagged as missing.

Model strategy: Gemini 1.5 Flash via DSPy/LiteLLM is the current workhorse and is fine for high-volume drafting. The DSPy/LiteLLM routing means we can send higher-stakes steps (final procedure generation, deviation root-cause, an LLM-judge in eval) to a stronger model — e.g. Claude Sonnet/Opus — without rewriting the agents. Decide per task by cost vs. consequence; keep embeddings consistent with whatever dimension the store is fixed to.

Step 4 · evaluation

No role ships until it passes its bar

Grounded agents fail quietly — a plausible citation to the wrong clause looks fine until an auditor checks. Every role gets a golden set and hard acceptance bars before it goes in front of a customer.

🎯

Golden Q&A per role

~50 representative tasks per role with known-correct, human-verified cited answers (drawn from real design-partner projects). The regression set the agent must keep passing as the corpus and prompts change.

🔗

Citation faithfulness

For every claim, does the cited chunk actually support it? Automated groundedness scoring (LLM-judge + spot human audit) — the single most important metric, because it's the trust promise.

🚫

Refusal calibration

Feed out-of-corpus and adversarial questions; the agent should refuse, not invent. Measure false-answer rate on unanswerable prompts and tune the confidence threshold.

🧪

Red-team for hallucination

Deliberately ambiguous specs, contradictory documents, missing sequences. Target: zero unsupported claims reach a human as if they were grounded.

Build the harness once, reuse per role. The eval set plugs into the existing agent_runs audit data — every run already records input, output, citations, refusals, token cost, and latency. That telemetry is the evaluation substrate; we add labels and judges on top, and gate each role's rollout on the bars above.

Step 5 · data acquisition & licensing

What we can ingest — and the copyright line we don't cross

"Free to read" is not "free to embed in a product corpus and resell answers from." Grounding a commercial agent makes licensing a first-class concern. The rule of thumb: U.S. government works are public domain and safe; the customer's own documents are theirs to let us use; copyrighted standards are cite-only unless licensed.

The clean architecture this implies: the agent can be deeply standards-aware without ever storing copyrighted standard text — it grounds on public-domain federal guides that teach the same process, cites the standard clause by number, and surfaces the relevant section of the customer's own licensed documents. That's both legally clean and exactly the citation-trust behavior buyers want. It's the same line we hold on the Study Guide: use the free legitimate corpus, point to (don't pirate) the paid standards.

The rollout

Sequence it — corpus first, one role deep, then fan out

Don't build six agents at once. Harden the foundation, prove one role end-to-end against the bars, then replicate the pattern role by role.

Phase 0

Corpus + pipeline hardening

foundation

Assemble Layer A from the public-domain sources (PNNL/GSA/FEMP + ACG templates), tag source tiers, and fix the pipeline: reconcile embedding dimensions, switch to query-embedding retrieval, structure-aware chunking, and corpus versioning. Nothing role-specific yet — make grounding trustworthy first.

Phase 1

One role, end-to-end

prove it

Take the Cx Engineer (the current procedure-drafting agent already lives here): add the role persona + scoped retrieval, stand up the eval harness and golden set, and hit every acceptance bar on a real design-partner project. This is the template every other role copies.

Phase 2

Fan out the roles

replicate, eval-gated

Add OCA → Field Technician → Construction Manager → Design Engineer → Owner/FM, one at a time. Each reuses the Phase-1 pattern and must clear its own golden-set bars before it's exposed. No role ships on vibes.

Phase 3

Close the lesson loop

the moat

Wire Layer C: every human accept/edit becomes org-scoped training signal that improves future retrieval and drafts. This is what makes each project faster than the last — the compounding asset behind the per-org pricing and the hardest thing for a competitor to copy.

The throughline: we already have the rare, hard part — a cited, refuse-when-unsure agent with a full audit trail. The work ahead is mostly knowledge and discipline: give the agents the right grounded corpus, scope them per role, and never let one ship without proving it's faithful. That's how the agents become genuine support for every seat in the commissioning process — and why earning the certifications and learning the body of knowledge ourselves directly sharpens the product.
